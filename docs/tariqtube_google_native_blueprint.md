# 🎬 TariqTube 2.0: Master Architectural Blueprint (v4.1)

## 📋 1. Core Architectural Principle: Multilingual Content Factory
TariqTube 2.0 is an enterprise-grade, **Project-Centric** and **Multilingual** content factory. The system is designed to produce consistent, episodic content at scale across multiple languages and social platforms from a single master production source.

### Key Pillars:
- **Projects as the Source of Truth**: All production state, character persistence, and episodic memory live at the Project level, decoupled from publishing channels.
- **Multilingual Synchronization**: One "Master Visual Episode" (Visual Anchor) is rendered once. Multiple "Language Variants" (Linguistic Anchors) are then generated by swapping narration and metadata.
- **Platform Agnostic**: The system is built to support YouTube, TikTok, Instagram, and more without restructuring code.

## 🏗️ 2. Detailed Data Model: System Entities

### A. Organizational Hierarchy
| Entity | Description | Ownership |
| :--- | :--- | :--- |
| **Workspace** | The top-level administrative container for the entire account. | L1 |
| **Project** | **The Master Owner.** Defines the visual style, audience, and Asset Registry. | L2 (Belongs to Workspace) |
| **Series** | A collection of related content with sequential ordering (Seasons, Arcs). | L3 (Belongs to Project) |
| **Episode** | **The Production Anchor.** A single story beat with one visual master render. | L4 (Belongs to Series) |
| **LanguageVariant**| **The Linguistic Anchor.** Reuses the master video with localized audio/SEO. | L5 (Belongs to Episode) |

### B. Asset & Distribution Entities
| Entity | Description | Ownership |
| :--- | :--- | :--- |
| **Asset** | Persistent entities like **Characters**, **Environments**, and **Voice Profiles**. | Belongs to Project |
| **Content Unit** | Specific media file types (Full Episode, Teaser, Highlight Short, Recap). | Belongs to Variant |
| **Channel** | A publishing destination (YouTube Channel, TikTok Account). | Base Link |
| **Publishing Route** | Mapping logic (SEO templates, regional schedules) between Units and Channels. | Config |
| **Schedule Item** | A queued task defined by a timestamp and a specific LanguageVariant. | Belongs to Router |
| **Post Result** | The immutable record of a successful publish (External ID, URL). | Belongs to Variant |

## 🔐 3. LanguageVariant Implementation Detail
A **LanguageVariant** is the final localized product. It must track the following implementation-grade fields:
- **Linguistic Context**: `language_code` (e.g., `ar`, `en`).
- **Linguistic Assets**:
    - `translated_script`: The AI-localized version of the master script.
    - `voice_assignments`: Mapping of Character IDs to locale-specific Voice IDs.
- **Metadata (Localized SEO)**:
    - `localized_title`, `localized_description`, `localized_hashtags`.
    - `localized_subtitles`: (e.g., VTT/SRT paths).
- **Media Results**:
    - `master_video_reuse`: Pointer to the Episode's `visual_master.mp4`.
    - `localized_audio_path`: GCS path to the generated narration track.
    - `final_render_path`: GCS path to the multiplexed high-quality MP4.
- **Publishing State**:
    - `variant_status`: (Generating-Audio-SEO, Multiplexing, Ready-to-Publish, Published, Failed).
    - `channel_target_id`: Pointer to the specific language channel.
    - `external_post_results`: Map of `{platform_id: external_id}`.

## 🚀 4. Platform-Agnostic Publishing Router
The system is built to be multi-platform by design:
- **Router Logic**: Receives a `Content Unit` and its `Publishing Route`. It identifies the target platform (YouTube, etc.) and executes the corresponding **Platform Adapter**.
- **Adapter Model**: New platforms (Instagram, TikTok) are added as standalone adapters without touching the core Episode/Project logic.
- **Targeting**: An English variant can be pushed to a YouTube channel while the Arabic variant of the same episode is pushed to a TikTok-Arabic account.

---

## 🎨 5. Project-Centric Ownership Model
Production logic is strictly hierarchical:
1.  **Project Registry**: Stores immutable character visual seeds and vocal signatures.
2.  **Episodic Memory**: Gemini 1.5 Pro reads the previous 5 episode summaries from the Project state to ensure narrative continuity.
3.  **Synchronization**: Episode `45` in Arabic has the same numbering and visual content as Episode `45` in English.

## 💾 6. Concrete System Infrastructure

### Firestore Schema Specification
- `projects/{pj_id}/`
    - `registry/assets/`: {type, visual_seed, voices: {ar: id, en: id}, versions: []}
- `series/{sr_id}/`
    - `episodes/{ep_id}/`
        - `metadata`: {order, visual_status, master_video_path, story_approved: bool}
        - `variants/`: (sub-collection)
            - `{locale_id}`: {status, script, audio_path, render_path, seo_meta, publish_log: []}
- `publishing/routes/`: {project_id, lang_code, platform, channel_id, schedule_pattern}
- `publishing/schedule/`: {timestamp, variant_path, unit_type, status}

### Cloud Storage (GCS) Hierarchy
- `tariqtube-production/`
    - `projects/{pj_id}/assets/` (Master Character Ref Sheets)
    - `episodes/{ep_id}/`
        - `visual_master.mp4` (High-bitrate, no narration)
        - `renders/`
            - `ar/` (full_ar.mp4, teaser_ar.mp4, short_ar.mp4)
            - `en/` (full_en.mp4, teaser_en.mp4, short_en.mp4)
        - `audio/` (Localized narration tracks)
        - `metadata/` (Localized JSON metadata packs)

## 🕒 7. Episode Control & Scheduling
The system monitors status across the entire pipeline:
- **Imported**: Episode pack received from Series source.
- **Generated**: Visual Master and all local Language Variants are multiplexed.
- **Published**: Video ID and URL are verified and stored.
- **Next-Up Logic**: The Series Manager identifies the next production unit based on the `last_produced_index`.
- **synchronized Cadence**: Teasers and Highlight Shorts are automatically inserted into the publishing queue in the gaps between the master episode drops.
